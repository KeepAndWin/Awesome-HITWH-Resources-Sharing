第六章 进程同步

一、背景

竞争条件：

多个进程并发访问和操作同一数据，且执行结果与访问发生的特定顺序有关。 竞争条件的结果是不可预测的。

二、临界区问题

  • 临界区问题
    • 多个进程同时操作共享数据时，每个进程拥有操作共享数据的代码段（程序段），而这代码段称为临界区(critical section)。
    • 如共享变量、共享表、共享文件等
  • 解决竞争条件问题的关键
    1. 确保单个进程在临界区内执行
    1. 确保其他进程也可以进入临界区
  • 临界区问题的通用结构：

    进入区、临界区、退出区、剩余区

  • 解决临界区问题需要满足如下条件
    1. 互斥（Mutual Exclusion ） – 某一个进程进入了临界区，其他进程就不能进入
    1. 前进（Progress） – 如果没有进程在其临界区内执行，并且有进程需要进入临界区，那么只有那些不在剩余区内执行的进程可以参加选择，以便确定谁能下次进入临界区，而且这种选择不能无限推迟
    1. 有限等待（Bounded Waiting） – 一个进程做出进入临界区的请求，直到该请求被允许，必须有限等待

三、Peterson's 算法

适用于两个进程

  • 设置两个变量 int turn //表示哪个进程可以进入其临界区（竞争） boolean flag[2] //表示哪个进程想要进入其临界区（true）
  • 前提条件是加载和存储指令是原子指令，即加载和存储指令是不可被中断的指令。

    代码实现：i——当前进程，j——另一个进程

    do {
	flag[i]=true;
	turn=j;
	while (flag[j] && turn==j); //j想要进入临界区且j可以进入临界区
	//临界区
	flag[i]=false;
	//剩余区
}while(true);

四、硬件同步

  • 硬件方法解决：许多系统都拥有简单硬件指令，并描述如何用它们解决临界区问题
  • 现代计算机系统提供特殊指令叫原子指令——不可中断的指令
    1. TestAndSet（）：检查和设置字的内容
    1. swap（）：交换两个字的内容不可中断的指令
  • TestAndSet() 指令&swap()指令：
    • testandset:检查和设置字的内容。将参数target 置true，并将原值作为返回值返回。
    • swap：交换两个字的内容参数的值
    boolean TestAndSet (boolean *target)
{
	boolean rv = *target;
	*target = true;
	return rv;
}

    声明一个布尔全局变量lock, 初始化为lock=false 另外，每个进程也有一个局部变量key, 初始化为true.

    当lock或key 为false, 便可进入临界区

    方案一,testandset
// lock：厕所外的锁
do {
		while(TestAndSet(&lock)); //如果锁是开的就能进去，并且进去之后把锁锁上（独占厕所）
			//临界区
		lock=false; //用完之后把锁打开，让别人可以用
			//剩余区
}while(true);

===================================================================================
方案二,swap
// key: 打开厕所外面的锁
// lock：厕所内反锁
do {
		key=true;
		while(key==true) Swap(&lock,&key); //如果有钥匙，且厕所没有反锁，就能进去（双锁破解）
																			 //进去之后反锁（独占厕所）
			//临界区
		lock=false; //用完之后把内部锁打开，外面的锁有钥匙的就能打开
			//剩余区
}while(true);
    • 存在的问题：

      Peterson`s 算法、TestAndSet( ) 和swap( ) 原子指令(atomic instruction)操作会存在忙等待的问题（busy waiting）

      • while(TestAndSet(&lock))
      • while(swap(&lock, &key))
  • 有限等待原子指令——解决无限等待的问题，不是忙等待的问题

    声明一个布尔变量waiting确保有限的等待

    1. 局部变量key，初始化为true，每个进程都有；
    1. 全局变量lock，初始化为false;
    1. 用变量waiting[i] 表示等待进入临界区的进程，初始化为false;

    基本思想

    do {
		waiting[i] = true; 
		key = true;
		while(waiting[i] && key) key = TestAndSet(&lock); //看看内锁有没有打开，打开了就进去
		waiting[i] = false; //不等了
		// 临界区
		j = (i + 1) % n;
		while(( j != i) && !waiting[j]) j = (j + 1) % n; //假设这个j不在等待找下一个直到全找到
		if (j == i) // 没有找到，开全局变量内锁
		lock = false;
		else  // 找到了，不让它等待了
		waiting[j] = false;
		// 剩余区
}while (true);
    1. i 表示当前运行的进程
    1. n 表示进程的数量
    1. j 表示 i 进程下一个进程 检查下个进程 j 是否要进入临界区
      1. 要是waiting[j] == false，即不想进入临界区或已经进入临界区，那么跳过，检查下一个进程；
      1. 要是waiting[j] == true，即想进入临界区， 就允许进入，并把 waiting[j] 设置成false;
      1. 检查了一遍，没有进程想进入临界区，就把lock设置成false

五、信号量

  1. 临界区问题的三个解决方案（Peterson 算法，硬件同步，有限等待原子指令）均是单资源 的同步操作（因为只有一个临界区），如果想解决多资源的同步操作只能通过信号量
  1. 适用于单个或多个资源的同步操作
    1. 用S来表示信号量，S是整数变量
    1. 只能通过标准原子操作来访问信号量
      • wait(S) operation : P(S)→S--
      void wait(S){
		while (S<=0);
		//无操作
		S--;
}
      • signal(S) operation : V(S)→S++
      void signal(S){
		S++;
}
  • 二进制信号量：

    适用于单资源的共享， mutex值为资源数量，初始化为1

    信号量的值只能为 0 或 1

    算法描述：

    • 想要进入临界区的进程先通过wait操作看一下有没有可用的资源。若资源的数目＞0，就占用一个资源，进入临界区。否则就原地等待。
    • 用完资源之后调用signal来释放资源，资源数目+1
    do{ 
		waiting(mutex);
		 //critical section 
		signal(mutex); 
			//remainder section 
} while (true);
  • 计数信号量(counting semaphore)
    • 适用于多资源共享，共享资源的数量为 n
    • wait(n) 操作为减，signal(n) 操作为加，当n为0时表明所有资源都被占用
    • 信号量的实现： 信号量的实现关键是保障wait()和signal()操作的原子执行，即必须保障没有两个进程能 同时对同一信号量执行wait()和signal()操作。
    • 保障方法： 单处理器环境下，禁止中断; 多处理器环境下，禁止每个处理器的中断。但这种方法即困难又危险
  • 无忙等待的信号量
    1. 问题：忙等待（busy waiting），自旋锁（spinlock）

      为了解决忙等待的问题，让忙等待的进程挂起（blocking），可以进入临界区时，让进程重新启动（wakeup）

      • 挂起: 进程从运行状态（running）转换成等待状态（waiting)
      • 重启: 进程从等待状态转换成就绪状态
      • 当一个进程位于临界区，其他进程必须执行不必要的循环操作，浪费CPU。这种类型的信号量称为自旋锁。
    1. 无忙等待的信号量实现: 我们将信号量定义如下：
      typedef struct { 
	int value; //是整数值，是资源数量 
	struct process *list; //处于等待状态的进程队列
} semaphore
      1. 定义信号量结构体，整型值 value 表示资源数，进程类型数组 list 表示被挂起的进程队列；
      1. 调用 wait()时，先修改可用资源数，当修改后的资源数小于 0 时，表征资源数不足，将本进程加入挂起队列，阻塞进程的执行；
      1. 调用 signal()释放资源时，先修改可用资源数，当显示的资源数还是不足时，唤醒挂起队列里面被挂起的进程；
      1. 唤醒进程的操作，可以让已经在挂起队列里等待的进程，跳过和其他发起请求的进程竞争资源的过程，得到执行的机会，防止其长时间得不到响应；
      1. 这种挂起的操作，让 CPU 在执行的进程阻塞的时候没有忙等待；

    1. 死锁
    1. 饥饿

六、经典同步问题

  1. 有限缓冲问题:

    假定缓冲池中有 n 个缓冲项, 每个缓冲项能存一个数据项

    1. 当缓冲池满的时候，不能写（full）
    1. 当缓冲池空的时候，不能读（empty）
    1. 读写互斥

    生产者-消费者问题中的有限缓冲、读写互斥、写写互斥、读读互斥问题。

    生产者：写

    while (true)  {
		//   produce an item
		wait(empty);//empty>0, 只要有空缓冲项，就写
		wait(mutex);
		//  add the item to the  buffer
		signal(mutex);
		signal(full);
}

    消费者：读

    while (true) {
		wait(full); //full > 0, 只要缓冲项里有数据，就读
		wait(mutex)
		//  remove an item from  buffer
		signal(mutex);
		signal(empty);
		//  consume the removed item
}
    • 信号量empty 表示空缓冲项个数（为0 时不能生产）有限缓冲
    • 信号量full 表示满缓冲项个数（为0 时不能消费）有限缓冲
    • 二进制信号量mutex 实现生产与消费互斥 读写互斥

    流程：

    • 生产者先检测空内存和共享内存互斥状态，写操作完成时，释放互斥状态并更改 full 变量
    • 消费者先检测满内存和共享内存互斥状态，读操作完成时，释放互斥状态并更改 empty 变量
  1. 读者-写者问题: 多写者、多读者的写写互斥、读写互斥、记录读者数量问题。
    • 写写、读写互斥： 信号量wrt（读者写者共享，初始化1）实现写操作与其他操作互斥
    • 记录读者数量： 信号量mutex 实现readcount（初始化0）更新的互斥

    读者：每次修改readcount之前，加锁mutex即可

    while(true){
	wait(mutex);
	readcount++;
	if (readcount==1) //第一位
		wait(wrt);
	signal(mutex); //读不是互斥的，所以可以释放，让下一个读者可以读
	//reading
	wait(mutex);
	readcount--;
	if (readcount==0) //最后一位结束
		signal(wrt);
	signal(mutex);
}

    写者：

    while(true){
	wait(wrt);
	//writing
	signal(wrt);
}

    流程：

    1. 先验证读者计数变量互斥修改的信号量 mutex 的可写性(该变量不可用表示有进程正在修改 readcount)；
    1. 修改读者计数变量后，若为第一个读者，则需验证读写状态信号量是否可读(即若正在进行写操作是不允许读的)；
    1. 释放读者计数变量互斥修改的信号量 mutex；
    1. 进行读操作(此时的读操作不是互斥的，可能有多个进程在读)；
    1. 完成读操作后，再次锁住读者计数变量互斥修改的信号量 mutex，修改读者数，修改完成后查验读者数。若读者数信号量不为 0 即仍有读者进程在读共享内容，则不可释放读写信号量 wrt；若读者数信号量为 0，则本进程为最后一个读者，则释放读写信号量 wrt；
    1. 释放读者计数变量互斥修改的信号量 mutex
  1. 哲学家就餐问题：

    5 个哲学家围绕一张圆桌而坐，桌子上放着 5 支筷子，每两个哲学家之间放一支；哲学家的动作包括思考和进餐，进餐时需要同时拿起他左边和右边的两支筷子，思考时则同时将两支筷子放回原处。如何保证哲学家们的动作有序进行?

    共享资源是两人之间的筷子，因此某个哲学家(进程)进餐时，要验证其左右两边的筷子(临界量)是否可用，并且在其用餐期间锁死其两边的筷子(临界量)是多个进程之间多个资源共享的问题

    多进程多资源共享问题，出现死锁。

    需要引入管程

七、管程

  1. 管程：封装数据和一组互斥操作，并保证同时最多只有一个线程执行管程代码。在此问题中，哲学家i（thinking，hungry，eating）只有在其相邻两个邻居都不进餐时才能拿起筷子。
  1. 管程的实现2方法：
    • 利用抽象化概念
    • 利用信号量
  1. 管程的概念：

    管程是一种用于多线程互斥访问共享资源的程序结构。

    • 采用面向对象方法，简化了进程间的同步控制
    • 任意时刻最多只有一个线程执行管程代码
    • 正在管程中的线程可临时放弃管程的互斥访问，等待事件出现时恢复。
  1. 引入管程的原因
    1. 把分散在各个进程中的临界区集中起来进行管理
    1. 防止进程有意或无意的违法同步操作（交换wait()和signal()操作顺序；用 wait()替代signal()操作；省略了wait()或signal()操作）
    1. 便于用高级语言来书写程序，便于程序正确性验证。
  1. 管程的使用
    1. 在对象/模块中，收集相关共享数据
    1. 定义访问共享数据的方法
  1. 管程的组成
    1. 一个锁：控制管程代码的互斥访问
    1. 零或多个条件变量：管理共享数据的并发访问（条件变量的数目即共享资源/ 数据的数量）
    1. 一个条件变量对应于一个等待队列，每个条件变量仅有wait()和signal()操作
  1. 条件变量: 当调用管程过程的进程无法运行时，用于阻塞进程的一种信号量
    进程请求资源后两种情况： 1.若无资源可用，则把其挂起，放到该资源对应的等待队列 2.若有资源可用，则把该对应等待队列的资源分配给该进程

8. 哲学家就餐问题：

  • 要求一个哲学家在两只筷子均可用时才拿起筷子； (𝑠𝑡𝑎𝑡𝑒[(𝑖 + 4)%5]! = 𝐸𝐴𝑇𝐼𝑁𝐺) 𝑎𝑛𝑑 (𝑠𝑡𝑎𝑡𝑒[(𝑖 + 1)%5]! = 𝐸𝐴𝑇𝐼𝑁𝐺)
  • 区分哲学家所处的三个状态：THINKING, HUNGRY, EATING；
  • 声明条件变量-哲学家 condition self[5]， 对条件变量仅有的操作是wait() 和 signal()；

声明管程类 DP： 定义记录五个哲学家的状态的枚举类型数组 state，枚举类型比普通代码声明更加节省内存

monitor DP{
		enum{THINKING,HUNGRY,EATING} state[5]; 
		condition self[5]; //定义条件变量数组 self 

		void pickup(int i) //第 i 位哲学家想要进行进餐，即进程 i 想要进入临界区操作
		{
			state[i] = HUNGRY; //先修改哲学家(进程)的状态为饥饿状态，即提前想要进入临界区的申请
			test(i); //调用 test 方法检测哲学家(进程)需要使用的资源的可用性
			if (state[i]!=EATING) self[i].wait(); //若不可用则使用条件变量 condition 阻塞自身进程
		}

		void putdown(int i) //第 i 位哲学家(进程)完成共享资源操作
		{
			state[i] = THINKING; //更改自身的状态 state，
			//调用 test 方法检测其之前占用的资源是否有进程在等待.若有则唤醒阻塞的进程
			test((i+4)%5); //左边的
			test((i+1)%5); //右边的
		}
		
		void test(int i) //对提出申请的第 i 位哲学家(进程)检测其左右的资源是否可用
		{
			//可用的情况下更改自身进程state，锁定 condition
			if ((state[(i+4)%5)!=EATING)&&(state[i]==HUNGRY)&&(state[(i+1)%5)!=EATING)){
					state[i] = EATING;
					self[i].signal();	
			}
		}
		//初始化状态，所有的人都在thinking
		void initialization_code(){
				for (int i=0;i<5;i++){
					state[i] = THINKING;
				}
		}
}
